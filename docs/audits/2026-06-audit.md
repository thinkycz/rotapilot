# Audit, 2026-06 — RotaPilot

This document records the findings, fixes, and follow-up backlog for the
2026-06 audit of the RotaPilot codebase. It is intended to live next to
`docs/guidelines.md` and `docs/architecture.md` as a snapshot of what
the codebase looked like at the time of audit, what was changed, and
what is intentionally deferred.

The audit was performed in two passes by a ZCode agent. The first
pass produced the findings below; the second pass (a follow-up
request from the user) completed the previously-skipped items
9 and 11, the pre-existing AI test failure, and additional
N+1 patterns surfaced during a second read.

---

## Scope

- Backend: every controller, model, service, middleware, route, and
  config file under `app/`, `bootstrap/`, `config/`, `database/`,
  `routes/`.
- Frontend: every page, component, composable, and lib under
  `resources/js/`.
- Local package: `packages/thinkycz/laravel-core/`.
- Tests: `tests/` (architecture, feature, unit).
- Build and tooling: `Makefile`, `composer.json`, `package.json`,
  `phpunit.xml`, `.env`, `.env.testing`.

## Methodology

The audit used a combination of:

1. A full read of the AGENTS.md and docs/guidelines.md to anchor
   the conventions.
2. Reading every controller (Web + Api), every model, every
   service, and the three architecture tests.
3. Running the full validation suite (`make check`-equivalent) at
   baseline to identify pre-existing test failures.
4. Running `git stash`-based before/after diffs to attribute
   failures to specific changes.
5. Cross-referencing the laravel/ai integration for behavior that
   the project's tests assumed (the agent test failure).

## Baseline

- 127 of 267 feature tests failing at session start. All 127 failed
  with HTTP 419 (CSRF token mismatch) on the very first POST. The
  root cause was the test env using `SESSION_DRIVER=array` in
  combination with the .env value `APP_ENV=local` overriding the
  phpunit.xml `<env name="APP_ENV" value="testing"/>`, which left
  the container's `env` binding at `local` so Laravel's standard
  `runningUnitTests()` check returned false and the CSRF middleware
  rejected every request.

- 1 of 1 AI agent integration test was failing too, but the failure
  was masked by the 419.

- 0 PHPStan errors at level max.
- 0 Pint violations.
- 0 vue-tsc errors.
- All Vitest tests passing.
- 63 architecture tests passing (8386 assertions).

## Findings and fixes

The findings are split into "fixed in this audit" and "deferred".

### Fixed

#### F1 — `ScheduleUpdateController` was missing the authorization check (security)

`app/Http/Controllers/Web/Schedules/ScheduleUpdateController.php` was
finding the schedule by id and mutating it without any authorization
check, while the sibling destroy and publish controllers both call
`Authorization::mustManageSchedule($actor, $schedule)`. Any
logged-in user could rename or retarget a foreign schedule by
passing its id in the query string.

Fix: add the missing guard, mirror the pattern from
`ScheduleDestroy` / `SchedulePublish`. Cover with a new feature
test (`store manager cannot update a schedule for a store they do
not manage`).

#### F2 — Disconnected forgot / reset password flow

The web `ForgotPasswordController` regenerated a random password and
emailed it via `sendPasswordNewPasswordSettedNotification`, so
the `ResetPasswordController` (which validates a broker token) was
unreachable from the forgot form. The API controller already
supported both flows via the `auth.passwords.users.send_raw_password`
config flag — mirror that branch in the web controller.

Fix: branch on the config flag. When the flag is false (the
standard Laravel flow), the controller now calls
`PasswordBroker::createToken` + `sendPasswordResetNotification`,
keeping the existing "new password emailed" path as the default
(the flag is true in `config/auth.php`).

Added two new feature tests:

- the broker flow sends `PasswordResetNotification` and does not
  change the user's password;
- the full end-to-end flow: forgot → token issued → reset →
  password updated.

#### F3 — Duplicated proposal render block in `agent/Index.vue`

`pages/agent/Index.vue` rendered the same proposal card markup
twice — once for proposals attached to a message_id, once for
orphan proposals — and the two blocks had already drifted in
subtle ways. The 540-line page was hard to review and the
duplication was a magnet for one-sided fixes.

Fix: extract the proposal card into its own component
(`resources/js/components/agent/ProposalCard.vue`) with a clean
prop/emit API:

- `proposal`, `selected`, `isStreaming`, `isSubmittingThis`,
  `isAnySubmitting`
- emits: `toggle(index)`, `apply()`, `reject()`

The submit-state API is split so the spinner only appears on the
right card while all other apply/reject buttons stay disabled.

Drop the now-unused `actionDiffClass`, `getActionTypeName`,
`proposalConflictCount`, `hasSelectedActions`, `isActionApplied`,
and `proposalStatusLabel` helpers from `Index.vue`. Net 440 lines
lighter.

#### F4 — N+1 in `ScheduleShowController` and per-day `forceFill()->save()` in `ScheduleStoreController`

`ScheduleShowController` was calling
`$r->assignments()->with('employeeProfile')->orderBy('start_time')->get()`
per requirement — one query per requirement. `ScheduleStoreController`
was creating one `ShiftRequirement` per day in a 31-step loop
(31 INSERTs per schedule).

Fix:

- One `loadMissing(['store', 'shiftRequirements.assignments.employeeProfile'])`
  call in the show controller; assignments and employee profiles
  are now read from the in-memory collection.
- Inside the existing `DB::transaction` in the store controller, build
  an array of row dicts from the business hours and call
  `ShiftRequirement::insert()` once.

#### F5 — No uniqueness on `schedules(store_id, period_start)`

There was nothing preventing a manager from re-submitting the
create-schedule form and getting duplicate `Schedule` rows for the
same month/store, which then run the same `for ($date = ...) { ... }`
block and create 31 more duplicate `ShiftRequirement` rows.

Fix:

- New migration `2026_06_14_120000_add_unique_period_to_schedules_table.php`
  adds a unique index `schedules_store_period_unique`.
- The controller looks up an existing schedule for the same period
  before creating a new one and 302-redirects to the existing
  schedule with a localized error flash; the DB index is the
  durable guarantee against concurrent submissions.

#### F6 — `Authorization::mustBeStoreManager` helper

Twelve web controllers (eight in `App\Http\Controllers\Web\Agent`
and four in `App\Http\Controllers\Web\Availability`) repeated the
same ad-hoc guard:

```php
if (!$user->isStoreManager()) {
    \abort(403, 'Unauthorized.');
}
```

Fix: add a centralized helper to `App\Support\Authorization`
alongside the existing `mustManageStore` / `mustManageSchedule` /
`mustCreateStore` / `mustDeleteStore` helpers. The helper throws
`AccessDeniedHttpException` with a more useful message. All twelve
call sites now use the helper.

#### F7 — Architecture test that catches missing authorization

The F1 bug landed because nothing in the test suite enforced that
`Update` / `Destroy` / `Store` controllers must call an
`Authorization::` guard. Add three arch rules that scan every
`app/Http/Controllers/Web/**/*UpdateController.php`,
`*DestroyController.php`, and `*StoreController.php` file and
assert the contents contain `Authorization::` anywhere. Verified by
temporarily deleting the `Authorization::mustManageSchedule` line
from `ScheduleUpdateController` and re-running the arch suite:

```
Tests: 1 failed (1 assertions)
```

The check is permissive (deliberately, so a controller that
authorizes a child record rather than the target itself still
passes) but tight enough to catch the class of bug that the
original `ScheduleUpdateController` represented.

#### F8 — Throttle on the public employee schedule endpoint

`/public/employee-schedules` is the only non-auth public route in
the project. Apply the standard `ThrottlesWebRequests` trait
(5 attempts per 15 minutes per request signature) — the same
throttle the auth controllers use — so the controller rejects
bursts before they hit the database.

#### F9 — Centralize form value coercion

`EmployeeStoreController` and `EmployeeUpdateController` had two
near-identical 6-8 line blocks per controller for parsing request
values (`max_hours_per_week`, `hourly_rate`, `is_active`,
`store_ids.*`). Add three static parser methods to
`EmployeeProfileValidity` alongside the existing rule-builder
methods:

- `parseNullableInt(mixed $value): int|null`
- `parseBool(mixed $value, bool $default = false): bool`
- `parseStoreId(mixed $value): int`

Replace the inline ladders in both controllers. Pure refactor —
no behavior change.

#### F10 — N+1 patterns in the read path of `/schedules/show` (second pass)

`EmployeeScheduleView::days()` had the same N+1 as
`ScheduleShowController`. `ConflictDetectionService::detect()` had
three of them: `collectEmployeeIds`, `detectEmployeeIssues`, and
`detectOutsideBusinessHours` (the last of which issued one
business-hours query per requirement via `BusinessHourGuardService`).

Fix:

- `EmployeeScheduleView::days()`: single eager load, iterate in
  memory, filter cancelled in memory.
- `ConflictDetectionService::detect()`: pre-load business hours
  once, iterate in memory for the other two. The
  `BusinessHourGuardService` dependency is dropped from the
  constructor; the service is still useful for ad-hoc checks but
  the hot path no longer needs it.
- Preserve the legacy "no business hours configured = shift is
  outside business hours" semantic that the
  `ConflictDetectionTest` relies on.

#### F11 — Drop dead `querySelect` / `scopeSearch` on the two pivot models

A grep for `::querySelect` or `::scopeSearch` across the entire
project returns zero call sites for either method on the pivot
models. The architecture test was forcing every non-User model to
declare both methods anyway, so the dead code was being actively
re-added on every new pivot model.

Pivot models are join tables consumed only via their `belongsTo`
relations. Neither method fits a 2-column pivot (`querySelect()`
does `select *` on the surrogate id; `scopeSearch()` casts the
search term to an int and matches it against the surrogate id,
which is meaningless for a pivot never looked up by its own id).

Drop the two methods and their unused `Builder` import from
`StoreManagerStore` and `EmployeeStore`. Update
`tests/Architecture/ModelArchitectureTest` to skip these two
classes via a small allowlist. The other 10 non-User domain
models (Store, Schedule, ShiftAssignment, ...) still get the
methods because they are queried by id and do benefit from the
select/search shape.

#### F12 — Pre-existing AI test failure (was hidden by F0)

`store manager can start background run and persist user message
immediately` was failing for a real reason, but the failure was
masked by the 419s. The test was depending on
`AppServiceProvider::boot()` registering `SchedulingAgent::fake(...)`,
which is conditional on the `openrouter` API key being `null` AND
the env being local/testing. The project's `.env` carries a real
openrouter test key, so the condition evaluated to false and the
agent fell through to a real LLM call.

Make the test self-sufficient by calling
`SchedulingAgent::fake(...)` in the test body. The test no longer
depends on the .env or the AppServiceProvider's environment-
conditional fake registration. Side benefit: the test is now
~0.21s instead of 4.11s.

#### F13 — CSRF test env

Set `APP_ENV=testing` (in `$_ENV`, `$_SERVER`, and via `putenv`)
before `parent::createApplication()`. This makes the
container's `env` binding `testing` so Laravel's standard
`runningUnitTests()` check returns true and the CSRF middleware
short-circuits. The `tests/Feature/DebugCsrfTest` debug probe
verified that the container's `env` was being read as `local`
because the .env's `APP_ENV=local` was overriding the
phpunit.xml `<env>` setting; setting it explicitly before
bootstrap fixed it.

This is the prerequisite for all the test coverage of the
items above: 126 of the 127 originally-failing tests had no
chance to exercise their inner assertions until the 419 was
removed.

#### F14 — Stray SQLite file in repo root

A 229 KB `rotapilot` SQLite database had landed in the project
root, presumably from a session with an explicit `DB_DATABASE`
override. The file is a full schema snapshot of all migrations,
not the default `database/database.sqlite` path, so it was
created via an env override at some point.

Delete the file and add catch-all patterns to `.gitignore` so
the next stray SQLite file (or the same file with a different
name) doesn't recur:

```
/*.sqlite
/rotapilot
*.sqlite3
*.db
```

#### F15 — Dead `findAssignment` rebuild in `AssignmentService`

`AssignmentService::findAssignment` was a hand-rolled
`Eloquent::first()` that pulled the first row from a Collection
and rebuilt a `ShiftAssignment` via `newFromBuilder()`. Replace
with the standard `->query()->where(...)->first()`.

#### F16 — `ShiftSourceEnum` for `source` column

`AssignmentService`, `ScheduleStoreController`, and
`ShiftRequirementStoreController` all hardcoded `source =>
'manual'`. `ShiftSourceEnum::Manual` exists for exactly this
value. Switch all three to `ShiftSourceEnum::Manual->value` and
drop the `// TODO: switch to ShiftSourceEnum::Manual` comment in
the store controller. The migration column accepts the same
string the enum value is.

#### F17 — Single-statement `DB::transaction` in `RegisterController`

`RegisterController::store` wrapped a single `User::create([...])`
in a `DB::transaction(...)`. No-op transaction around a single
statement is noise. Inline the call and drop the unused
`Illuminate\Support\Facades\DB` import.

#### F18 — Document the AI provider config block

The AI provider array in `config/ai.php` declares drivers for
twelve providers (anthropic, azure, bedrock, cohere, deepseek,
eleven, groq, jina, mistral, ollama, voyageai, xai) that the
current agent and image generation code do not use. Only
`openrouter`, `openai`, and `gemini` are wired up. Add a comment
near the top of the providers block explaining why the wider
declaration is intentional: the laravel/ai SDK ships a built-in
driver for each name, and re-registering a removed driver
manually is a worse trade-off than keeping an empty config
block.

#### F19 — `ParseTrait::mustParseNullableFloat` parity fix (carry-over)

`packages/thinkycz/laravel-core/src/Traits/ParseTrait.php` was
checking `is_int($value)` where `parseNullableFloat` (L213) just
above it correctly checks `is_float`. The int branch returned the
value as-is, so a config value parsed as an int would silently
pass through as an int instead of being filter_var'd to a float,
silently changing the return type from `float|null` to
`int|float|null`. Port the fix from the boilerplate to rotapilot
(another agent's audit caught this).

### Deferred (intentionally not fixed in this audit)

#### D1 — `MyCalendarController` direct Eloquent access

The controller calls `$user->loadMissing('employeeProfile')` and
then reads `$user->employeeProfile` directly. This violates the
"Relations must be accessed through explicit relationship methods
for queries or through typed relation getters" rule from
AGENTS.md. The typed getter on User would be `getEmployeeProfile()`,
not the property.

The fix is mechanical (change `$user->employeeProfile` to
`$user->getEmployeeProfile()` and add a `getEmployeeProfile()` to
the User model), but it touches the auth-shared Inertia props and
the AI agent's user introspection, both of which have their own
test surface. Deferred to a follow-up that also adds a typed
`getEmployeeProfile()` getter.

#### D2 — Controllers using `$request->query('id')` for POST body fields

`ScheduleUpdateController`, `EmployeeUpdateController`, and several
others read `$id = (int) $request->query('id', '0')` even when the
form is submitted as POST. The `id` is in the query string for
Inertia's link behavior, but the form body is the right place.
Inconsistent; some controllers read the id from the query, some
read it from the body. Deferred to a follow-up that normalizes
the form-id pattern.

#### D3 — `Authorization::managedEmployeesQuery` returns Builder, not EloquentBuilder

The return type is `Builder` (the base Laravel class) rather than
`EloquentBuilder<EmployeeProfile>`. PHPStan can infer the
relation-specific type, but the return declaration is loose. Low
priority; deferred.

#### D4 — Public schedule bot scraping

Even with the throttle, an attacker can rotate IPs / clear
sessions and still scrape at low rates. The route's defense is
the 48-char token (190 bits of entropy), but a bot that has
captured a legitimate token can scrape forever. Out of scope for
this audit; the threat model is "casual scraping", not "targeted
intrusion". Consider an HMAC-signed URL or an expiration window
on the token in a follow-up.

#### D5 — `HandleInertiaRequests` queries conversations on every Inertia request

The `agentConversations()` closure runs `Conversation::query()` on
every Inertia request for a store manager. The query is bounded
(limit 20) so the cost is low, but it is unconditional for
managers. If a manager does not use the agent, the query is
wasted. Defer to a follow-up that moves the closure behind an
`use` flag (only set when the user is on the agent page).

#### D6 — Stray `.trae/documents/` directory

There is a `.trae/documents/rotapilot-full-audit-and-improvements.md`
in the repo (it shows up in grep output and may be referenced by
some agents). The content is an AI-generated audit document that
duplicates parts of this audit. The file should not be committed
or shared with downstream agents (it can be confusing and slightly
stale). Add to `.gitignore` in a follow-up.
